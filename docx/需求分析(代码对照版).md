## 四、架构问题与风险分析

### 4.1 🔴 严重问题（阻碍系统正常运行）

| 编号 | 问题 | 影响 | 建议 | 状态 |
| ---- | ---- | ---- | ---- | ---- |
| ARCH-01 | **后端控制器架构偏差**：数据库设计为统一 `items` 表，但后端同时存在 `LostItemController`（`/api/lost-items`）和 `FoundItemController`（`/api/found-items`）两套独立控制器 | 数据不一致、维护成本翻倍、审核流程割裂 | 统一为 `ItemController`（`/api/items`），通过 `scene` 参数区分失物/招领 | ❌ **未解决** |
| ARCH-02 | **前端 API 路径不匹配**：前端 `items.js` 调用 `/api/lost-items` 和 `/api/found-items`，但进度文档定义的统一路径为 `/api/items` | 前后端联调失败 | 统一 API 路径设计 | ❌ **未解决** |
| ARCH-03 | **实体类重复与数据库设计不匹配**：`Item.java`（统一）与 `LostItem.java`（独立）同时存在，且 `LostItem.java` 使用独立枚举（`AuditStatus`/`LostItemStatus`），与 init.sql 的 stage/status 设计不符 | 代码混乱、编译冲突、数据映射错误 | 删除 `LostItem.java`，统一使用 `Item.java` + MyBatis-Plus LogicDelete | ❌ **未解决** |

### 4.2 🟡 中等问题（影响系统扩展性）

| 编号 | 问题 | 影响 | 建议 | 状态 |
| ---- | ---- | ---- | ---- | ---- |
| ARCH-04 | **分页响应格式不统一**：`FoundItemController` 返回 `ApiResponse<?>`，其他 Controller 返回 `ResponseEntity<ApiResponse<Page<T>>>` | 前端分页组件无法统一处理 | 统一为 `ResponseEntity<ApiResponse<Page<T>>>` | ⚠️ 待确认 |
| ARCH-05 | **Announcement vs Notice 命名混乱**：数据库表名为 `announcements`，但 `Announcement.java` 实体缺少 id 字段（与 BaseEntity 断开），`Notice.java` 是另一个独立实体，字段不同步 | 开发者困惑、CRUD 混乱 | 统一为 `Announcement.java`，继承 BaseEntity，字段与 init.sql 一致 | ❌ **未解决** |
| ARCH-06 | **管理员列表接口缺少分页**：`listUsers()`、`listReports()`、`listNotices()` 均返回全量 `List` | 数据量大时性能问题 | 增加分页支持，参考 `Page<>` MyBatis-Plus 模式 | ⚠️ 待确认 |
| ARCH-07 | **缺少文件上传接口**：init.sql 中 `item_attachments` 表设计完整，但后端无 `FileUploadController` | 无法完成图片附件功能，P0 优先级缺失 | 实现 `FileUploadController` | ❌ **不存在** |
| ARCH-08 | **SecurityConfig 权限配置待确认**：需确认 API 路径与 Spring Security 权限规则的匹配 | 可能存在越权风险 | 审查 SecurityConfig 配置 | ⚠️ 待确认 |

## 十、总体评估

| 维度 | 评分 | 说明 |
| ---- | ---- | ---- |
| 数据库设计 | ⭐⭐⭐⭐⭐ (95%) | init.sql 设计规范、索引完善、状态机清晰，与实际代码有偏差 |
| 后端实现 | ⭐⭐⭐☆☆ (55%) | 主要接口已实现，但存在 ARCH-01（架构偏差）和 ARCH-07（文件上传缺失）两个 P0 问题 |
| 前端实现 | ⭐⭐⭐☆☆ (60%) | 页面框架完整，核心工作在于 API 对接（失物/招领分离路径需调整） |
| 代码质量 | ⭐⭐☆☆☆ (45%) | 存在 ARCH-03（实体重复）、ARCH-05（公告命名混乱）、ARCH-09（备份文件）等多个问题 |
| 安全性 | ⭐⭐⭐⭐☆ (70%) | JWT + BCrypt + RBAC 基础完善，需审查 SecurityConfig 和权限注解 |
| 需求覆盖度 | ⭐⭐⭐☆☆ (55%) | 核心需求已定义，但 P0（统一物品 API + 文件上传）缺口影响系统核心流程 |

**总结**：

基于 init.sql 数据库设计的全面分析，项目存在**两个 P0 级别的架构问题**需要优先解决：

1. **ARCH-01 控制器架构偏差**：后端存在两套独立控制器（失物/招领分离），与 init.sql 统一 `items` 表设计严重偏差，必须合并为 `ItemController`
2. **ARCH-07 文件上传缺失**：后端无 `FileUploadController`，导致图片附件功能完全无法使用，是阻塞性缺陷

建议按 **Phase 1 → Phase 2 → Phase 3 → Phase 4** 顺序推进，Phase 1 的 4 个 P0 任务为阻塞项，必须优先完成。
