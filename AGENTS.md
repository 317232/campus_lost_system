# Project overview

- 这是一个基于 Spring Boot + Vue 3 的前后端分离的校园失物招领系统
- 项目结构：
  - backend/: Spring Boot 后端服务
    - 框架: Spring Boot 3.3.5, MyBatis-Plus 3.5.7
    - 数据库: MySQL
    - 安全: Spring Security + JWT
    - 邮件服务: Spring Boot Mail
  - frontend/: Vue 3 前端应用
    - 技术栈: Vite + JavaScript + Pinia + Vue Router + Axios
- 核心业务模块：用户管理、失物管理、招领管理、认领管理、公告管理、举报管理、管理员功能

# Function Description

- 用户管理：用户注册， 用户登录， JWT 身份认证，个人资料查看与修改，用户状态管理，角色与权限控制
- 失物信息发布：新增失物信息，修改本人失物信息， 删除本人失物信息， 查看本人失物发布记录， 上传附件图片，填写联系方式与丢失描述，
- 招领信息发布：新增招领信息，修改本人招领信息，删除本人招领信息，查看本人招领发布记录，设置或更新招领物品状态， 上传附件图片与联系方式
- 信息查询功能：失物/招领列表浏览，按名称搜索｜按类别筛选｜按地点筛选｜按时间排序｜分页查询， 详情页查看
- 认领申请：发起认领申请，填写认领说明，填写特征证明，查看个人认领记录，接收审核结果反馈，更新认领流程状态
- 管理员审核和后台管理：管理员审核与后台管理，审核失物信息，审核招领信息，审核认领申请，驳回、删除、下架违规信息，管理用户账号：发布与下架公告，处理举报/反馈，维护分类字典，查看操作日志
- 数据统计：通过数据字段后台汇总查询到前端展示，包括：失物数量统计，招领数量统计，认领申请数量统计，审核处理情况统计，用户发布情况统计

# Key Points
- 登录: 真实落地为登录使用邮箱进行登录，判断是否存在该邮箱用户，未注册人员跳转注册页面使用邮箱注册填写信息注册账户，注册人员可以使用邮箱进行登录
- 注册: 注册时需要填写邮箱，邮箱需要进行校验，校验通过后发送验证码，验证码通过后进行注册
- 找回密码: 找回密码时需要填写邮箱，邮箱需要进行校验，校验通过后发送验证码，验证码通过后进行重置密码
- 个人中心: 个人中心可以查看个人信息，修改个人信息，查看个人发布的失物信息，查看个人发布的招领信息，查看个人的认领申请，查看个人的操作


# Configuration

- 后端端口: 8080
- 数据库连接: MySQL (localhost:3306)
- 邮件服务: QQ 邮箱 SMTP (smtp.qq.com:465)
- JWT 配置: 使用 HS256 算法，有效期 2 小时

# Commands

- 后端启动: cd backend && mvn spring-boot:run
- 前端启动: cd frontend && npm install && npm run dev
- 后端打包: cd backend && mvn clean package
- 前端构建: cd frontend && npm run build

# Coding conventions

- 后端采用 Spring Boot + MyBatis-Plus 架构风格
- 前端采用 Vue 3 + Composition API 风格
- 错误处理统一使用 ApiResponse 格式
- 日志使用 SLF4J + Logback 进行记录
- 代码注释遵循 JavaDoc 和 JSDoc 规范

# Delivery

- 输出：代码 + 测试 + 变更说明
- 如果发现任务描述与代码现状冲突，先在结果里说明冲突点再实施

# Hermes Routing Policy

- 在这个仓库里，主模型负责需求澄清、业务边界、架构取舍、优先级排序、验收标准和最终拍板。
- 详细任务拆解、长上下文阅读、批量文件分析、可并行实现方案，优先交给 `route_task` 做显式分流；只有在 `route_task` 不可用时才退回 `delegate_task`。
- 只要满足任一条件，就必须委托子代理，而不是由主模型独自展开所有细节：
  - 涉及失物发布或认领流程，并且需要阅读 5 个以上文件
  - 需要拆成 3 个以上明确子任务
  - 详细实施步骤预计超过 10 条
  - 同时涉及前端、后端、SQL、接口契约中的两个及以上层面
- 子代理负责展开和执行细节，不负责最终决策；主模型必须汇总子代理结论后再给最终答复。
- 做失物发布、招领发布、认领申请、认领审核相关任务时，优先按“主模型先规划，子代理再细化”的顺序执行。
- 如果需要人工控场，优先使用会话命令：
  - `/route-status` 查看当前子代理
  - `/route-pause on|off` 暂停或恢复新委派
  - `/route-stop <subagent_id|all>` 中断运行中的子代理
  - `/route-run --preset publish-flow <goal>` 或 `/route-run --preset claim-flow <goal>` 手动启动显式分流
